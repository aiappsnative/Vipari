Quick start

Use the platform in this order

1 Repositories

Check whether the repo is connected, allocated, and onboarded. This is the first place to look if something feels missing.

 2 Dashboard

Review drift once a baseline exists. This is where current changes are compared to the approved reference point.

 3 Compliance

Use this after you understand the repo state. It is for evidence gaps, freshness, exports, and audit-readiness.

 4 Settings

Check workspace-level controls, billing, and feature access when the UI looks limited or unavailable.

 5 Agent integrations

Use this when you want to hand Vipari tools to an external AI agent through the customer MCP connector.

Why the name

Why Vipari

Vipari is named from viparinamadhamma, the idea that systems we treat as fixed are in fact subject to change.

That fits the product directly: prompts, tools, policies, and model routing all change, and Vipari exists to make those changes visible and reviewable.

Mental model

What Vipari is actually doing

GitHub repo
  -> Vipari discovers AI-relevant artifacts
  -> your team approves a baseline
  -> later changes are compared to that baseline
  -> reviewers decide whether the drift is acceptable

If there is no approved baseline, drift review will feel incomplete because the system has nothing trusted to compare against yet.

Common confusion

Connected is not the same as onboarded

Connected repo

Vipari can see the repository in workspace inventory.

Onboarded repo

Vipari has analyzed the repo and prepared it for baseline and drift workflows.

If a repo appears in inventory but not in review workflows, it is usually connected but not fully onboarded yet.

When you are stuck

Look for your situation, then go straight to the right page

I installed GitHub access but my repo is missing

Open Repositories and confirm the workspace can see the repo. Missing inventory usually means installation scope or connection state, not drift logic.

I can see the repo but I cannot review drift yet

Check whether onboarding finished and whether a baseline has been approved. Without that, the review flow is not fully grounded.

I see drift but I do not know whether it matters

Use the Dashboard first. Look for what changed, how broad the change is, and whether governance or evidence coverage also shifted.

The compliance area feels empty or stale

That usually means the underlying repo workflow has not produced fresh evidence yet. Check onboarding state, baseline status, and recent exports.

I do not understand a status label

Use the glossary below. The terms that matter most are connected, onboarded, baseline, drift, and freshness.

I think access is blocking me

Open Settings to verify plan limits, workspace-level controls, or role restrictions before assuming the repo data is broken.

Glossary

The concepts operators actually need

Workspace

Your organization container inside Vipari. It controls members, repo scope, plan access, and automation identities.

Baseline

The approved reference state. Later repo changes are judged against this, not against a generic policy document.

Drift

A meaningful difference between the approved baseline and the current repository state.

Fresh / stale / missing

These labels describe the confidence and recency of evidence. Missing means Vipari cannot confirm what it expected to find.

Prompts / tools / models / governance

These are the main artifact families Vipari monitors because they shape AI behavior and control coverage.

Onboarding

The setup phase where Vipari discovers artifacts, creates history, and prepares the repo for review workflows.

Page guide

Which page should I open?

  1. Repositories when you need to confirm visibility, onboarding, or whether a repo is in scope.
  2. Dashboard when you need to inspect drift and understand what changed.
  3. Compliance when you need evidence status, freshness, or export workflows.
  4. Settings when access, plan limits, or workspace controls may be affecting behavior.

Support expectations

What good help requests include

  • The workspace and repository involved.
  • What page you were on when you got stuck.
  • What you expected to happen.
  • What actually happened instead.
  • Whether the issue blocks review, onboarding, or compliance work.
